# Codex 专属规则

> 仅 Codex 读取（由 install.py 拼接进 ~/.codex/AGENTS.md）。通用规则见上方共享层。

## 编码风格

### 不可变性（关键）

始终创建新对象，不要原地修改。原理：防止隐藏副作用、便于调试、支持并发安全。

> 语言例外：Go 用指针接收器变更结构体是惯用写法。

### 文件组织

多个小文件 > 少量大文件。典型 200–400 行，最多 800 行。按功能/领域组织，不按类型。

### 错误处理

- 每一层显式处理。
- UI 层给用户友好消息，服务端记录详细上下文。
- **永远不要静默吞掉错误。**

### 输入验证

系统边界必须验证：用户输入、API 响应、文件内容、外部数据全部不可信。快速失败 + 清晰错误消息。

### 完成前检查

- [ ] 命名清晰、可读
- [ ] 函数 < 50 行
- [ ] 文件 < 800 行
- [ ] 嵌套 < 4 层
- [ ] 错误处理完整
- [ ] 无硬编码值
- [ ] 无变更（用不可变模式）
- [ ] 无硬编码密钥
- [ ] 用户输入已验证
- [ ] SQL 用参数化查询
- [ ] 无 console.log / 调试残留

## Git 工作流

提交格式：`<类型>: <描述>`，类型为 feat/fix/refactor/docs/test/chore/perf/ci。**不要加 AI 归属信息。**

PR 流程：
1. 分析完整提交历史（不只最新提交）
2. `git diff [base]...HEAD` 查全部改动
3. 起草摘要 + 测试计划（带 TODO checkbox）
4. 新分支用 `git push -u`

## 测试要求

- 最低覆盖率 80%
- 三种测试都要：单元 / 集成 / E2E
- TDD 工作流：RED → GREEN → REFACTOR → 验证覆盖率

## 研究优先级

实现新东西前的搜索顺序：
1. **GitHub 代码搜索** — `gh search repos` / `gh search code` 找现成实现和模板
2. **库官方文档** — 确认 API、版本特定细节
3. **包注册表** — npm / PyPI / crates.io，优先成熟库而不是手写
4. **网络搜索** — 前面都不够时再用

找到能解决 80%+ 的开源项目就 fork/移植/包装，不要从头写。

## 安全协议

发现安全问题：
1. **立即停止**
2. 修复关键问题再继续
3. 轮换任何可能暴露的密钥
4. 在整个代码库搜类似问题

强制检查：硬编码密钥、SQL 注入、XSS、CSRF、认证授权、速率限制、错误消息不泄露敏感数据。

## 代理编排

Codex 中只在当前环境明确提供子代理或相关工具时委托；如果没有对应代理，就按同等职责本地完成，不要假装已经调用外部代理。

### 可用代理映射

这些代理通常位于 `~/.claude/agents/`，在 Codex 中作为职责映射使用：

| 代理 | 用途 | 何时使用 |
|-------|---------|------------|
| planner | 实现规划 | 复杂功能、重构 |
| architect | 系统设计 | 架构决策 |
| tdd-guide | 测试驱动开发 | 新功能、bug 修复 |
| code-reviewer | 代码审查 | 编写代码后 |
| security-reviewer | 安全分析 | 提交前、安全敏感变更 |
| build-error-resolver | 修复构建错误 | 构建失败时 |
| e2e-runner | E2E 测试 | 关键用户流程 |
| refactor-cleaner | 死代码清理 | 代码维护 |
| doc-updater | 文档 | 更新文档 |
| rust-reviewer | Rust 代码审查 | Rust 项目 |

### 使用规则

无需额外提示即可应用这些职责：
1. 复杂功能请求 → 先做规划，必要时委托 planner。
2. 架构决策 → 使用 architect 视角检查边界、依赖和长期影响。
3. Bug 修复或新功能 → 使用 tdd-guide 视角，优先 RED → GREEN → REFACTOR。
4. 刚编写或修改代码 → 使用 code-reviewer 视角检查风险和测试缺口。
5. 安全敏感代码 → 使用 security-reviewer 视角，先处理关键问题。

独立任务可以并行时优先并行；任务之间有依赖时按依赖顺序推进。

## 被 Claude Code 委派时的协作契约

你经常被 Claude Code 当作「执行引擎」调用（通过 MCP `mcp__codex__codex` 或 `codex exec`）。分工固定：**Claude 负责规划与终审，你负责重活（读 repo、多文件改、跑测试、调查）**。被委派时遵守以下契约。

### 1. 沙箱自觉，绝不静默跳过

默认 `workspace-write` + 网络关闭。撞到权限边界时**不要默默放弃**，要在最终输出里显式列出「被沙箱挡下的操作」，让 Claude 决定是否预授权重试：

- 需要写 cwd 之外的目录 → 报告路径，提示需要 `writable_roots`。
- 需要联网（npm/pip install、curl、git fetch/push）→ 报告，提示需要 `network_access: true`。
- 不要把因权限失败的步骤当成"已完成"。

### 2. 固定汇报格式（让 Claude 能机械解析）

每次任务结束，按此结构输出：

1. **改动**：动了哪些文件，每个一句话变更摘要。
2. **验证**：跑了什么测试/构建，结果如何（贴关键输出，不贴全量）。
3. **未完成/不确定**：没做完的、需要 Claude 决策的、被沙箱挡下的。
4. 最后**单独一行** sentinel，便于脚本/Claude 检测完成：
   ```
   === CODEX DONE ===
   ```

### 3. 不扩大范围

只做被委派的那个任务。发现额外问题（无关 bug、死代码、可改进点）→ **写进「未完成/不确定」里报告**，不要顺手改。范围蔓延会让 Claude 的终审失效。

### 4. 风格服从目标 repo

匹配目标 repo 的现有风格优先于本文件的通用偏好；本文件其余编码规则（不可变、文件大小、错误处理、输入验证）仍然适用。
